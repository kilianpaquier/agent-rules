# Global instructions

## AI behavior

### Response

- English always.
- Concise. Short unless task need length.
- Don't explain what code do - well-named identifiers do that. Only non-obvious what/why.
- No follow-up task suggest unless asked.
- No unsolicited affirmation, closing pleasantry.

### Scope

- Never create doc files (README, CHANGELOG, etc) unless explicit ask.
- No git action (commit, push, PR) unless explicit ask.
- No unsolicited feature, refactor, abstraction. Do exact what asked.
- Prefer edit existing files over new.

### Process

- Follow instructions (project, file specifics, global) to letter.
- Ask before act when req/plan ambiguous, introduce refactor or anything structural.
- Before manual-edit file marked `Code generated by AI`, check if skill can regenerate from source schema/spec.
- Mandatory action → tool table (try primary first, fall back only if primary unavailable/insufficient; never skip straight to fallback):

  | Action                                                                           | Primary                                                                                                                                                                                     | Fallback                   |
  | -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------- |
  | Any library/framework/SDK/tool/CLI docs lookup (no vendor carve-out)             | `Context7` MCP                                                                                                                                                                              | "Raw" WebSearch/WebFetch   |
  | OpenTofu/Terraform provider, resource, module, datasource docs                   | `OpenTofu` MCP                                                                                                                                                                              | `Context7` MCP → WebSearch |
  | GitHub-hosted repo/issue/PR/pipeline/release object                              | `GitHub` MCP                                                                                                                                                                                | "Raw" WebFetch/WebSearch   |
  | GitLab-hosted repo/issue/MR/pipeline/release object                              | `GitLab` MCP                                                                                                                                                                                | "Raw" WebFetch/WebSearch   |
  | Source-code symbol search, call graph, architecture, dead code, structural query | `codebase-memory-mcp`/`codegraph` MCP, or their CLI below - complemented always by codebase language LSPs (`gopls`, `jdtls`, etc) for exact def/reference nav, not a fallback, use together | "Raw" file reads/grep      |

  CLI fallback, query-purpose commands only (setup/lifecycle commands like `init`/`index`/`sync`/`install`/`upgrade` excluded, not relevant to investigate-not-administer rule):

  ```sh
  codebase-memory-mcp cli <tool_name> '<json_args>'   # e.g. search_graph, trace_path, query_graph, get_architecture, get_code_snippet
  codegraph query <search>                            # symbol search
  codegraph explore <query>                           # relevant symbols' source + call paths
  codegraph node <symbol|file>                        # one symbol's source + callers, or read a file with line numbers
  codegraph callers <symbol>                          # what calls this
  codegraph callees <symbol>                          # what this calls
  codegraph impact <symbol>                           # blast radius of changing this
  codegraph files [path]                              # file structure
  codegraph affected [files...]                       # test files affected by changed source files
  ```

- Never assume/assert true, false, possible, impossible - re docs, code structure, external tools/frameworks/libraries/services, system behavior - without verify first.
- Challenge any approach - yours or user's - seem complex, indirect, reliant on workarounds, before implement.
  - Treat complexity as signal: investigate if simpler solution exist, present if found, confirm no direct alternative before implement.
- Prefer LSP ops over compile/build/lint commands.
- Never prototype in scratch/tmp files outside project - applies to final work product only.
  - Edit real target file direct, validate in place (prefer LSP then build/vet/test).
  - Harness-provided per-session scratchpad dir fine for genuinely ephemeral intermediates never the deliverable (e.g. extracting large tool result for `grep`).
- Never re-read file, run trivial check (JSON validity, file presence, etc) to confirm write/edit "worked" - only when tool result leaves outcome unclear.

## Code review

- Review only when user explicit ask - don't offer, suggest, initiate reviews.
- Sub-agent review: pass whole section as extra context.
- Review each modified file full (not only changed hunks) - catch inconsistencies, duplication, coherence issues.
- Focus:
  - Architecture, design: structure, abstraction boundaries, separation of concerns.
  - Bugs, functional correctness.
  - Security issues.
  - Adherence to project conventions.
  - Unnecessary complexity, duplication.
  - Documentation: missing or incorrect language-mandated comments (GoDoc, JSDoc, etc).
- Skip generated, vendored, third-party code (dirs: `vendor/`, `node_modules/`, `dist/`, `build/`; files with `Code generated by` / `DO NOT EDIT` / `@generated` headers). Strip those hunks from diff before pass to sub-agent.
- Flag dead/unreachable code as finding.
- Same focus for test files; check language-specific test conventions (e.g. `*.instructions.md` rules for package naming, helper markers, phase comments).

## Code style

- Comments, docs, commit messages: always English.
- Default no inline comments. Add only when WHAT/WHY non-obvious (hidden constraint, subtle invariant, bug workaround).
- Follow language doc standards on every exported identifier: GoDoc for Go, JSDoc for TypeScript.
- Never use regexes unless validated by user.
- Order fields/blocks alphabetically by default (struct fields, interface methods, HCL blocks, object keys, etc) unless language rule file define explicit order (e.g. GitLab CI job keys, Docker Compose service keys).

## Design

- No defensive checks for structurally impossible inputs/states. Always handle real error returns.
- Keep things straight, simple.
- No backwards-compat shims or stubs for removed code.
- No premature abstraction/interface. Use concrete type; add interface only when multiple implementations exist.

## Testing

- Test behavior, not implementation.
- No tests for trivial/unreachable code paths.
- Keep tests minimal, focused - one behavior per test.
- Mark test phases with `// Arrange`, `// Act`, `// Assert` comments (drop comment when phase has no steps).

### Tools

- Never use `curl`, prefer `WebFetch` tool.
- Use `git`, `node`, `python` only when explicitly asked.

## Language rules

- Language conventions: `~/.claude/rules/` (Claude Code) or `~/.copilot/instructions/` (Copilot).
- Extra project rules: `.claude/rules/` or `.github/instructions/`. Auto-loaded per file type.
